# 02. 제품 요구사항 문서 (PRD)

> 대상: 2~3일 해커톤 MVP
> 목표: "배포 전후 로그 drift 감지 → AI 요약 → 롤백 판단 신호" 전체 흐름을 동작하는 형태로 보여준다.

---

## 1. 문제 정의

엔지니어는 배포 후 "무엇이 달라졌는가"를 알 방법이 없어, 장애가 사용자에게 노출된 뒤에야 대응한다. LogLens는 배포를 기준으로 로그 패턴의 변화를 자동 감지하여 이 간극을 메운다.

---

## 2. 목표 / 비목표

### 2.1 목표 (In Scope)
- 배포 이벤트를 등록하고, 그 배포에 속한 로그를 수집(ingest)한다.
- 로그를 패턴으로 추출·정규화하고 통계를 집계한다.
- 직전 배포(baseline)와 현재 배포(current)를 비교해 drift를 계산한다.
- AI가 drift 결과를 자연어 리포트로 요약한다.
- 대시보드에서 배포별 drift 결과와 롤백 권고를 보여준다.

### 2.2 비목표 (Out of Scope — 해커톤 제외)
- 원본 로그 전체 저장 / 풀텍스트 검색
- 실시간 스트리밍 파이프라인 (Kafka 등)
- 멀티 테넌시, 조직/권한 관리(RBAC), SSO
- 알림 채널 다중 연동(해커톤은 대시보드 + 옵션으로 Slack webhook 1개)
- 자동 롤백 실행 (권고만 하고 실행은 사람이)

---

## 3. 핵심 사용자 시나리오 (User Stories)

### US-1. 배포 등록
> 엔지니어로서, 새 배포를 LogLens에 등록하여 이후 로그를 그 배포에 연결하고 싶다.
- 입력: 서비스명, 버전/커밋 SHA, 환경(prod/staging), 배포 시각
- 결과: `deployment` 레코드 생성, 고유 ID 발급

### US-2. 로그 수집(Ingest)
> 엔지니어로서, 배포 ID와 함께 로그를 보내면 LogLens가 패턴을 추출·집계해주길 원한다.
- 입력: 배포 ID + 로그 라인 배열 (batch)
- 처리: 패턴 정규화 → 패턴별 카운트/통계 집계 → 저장
- **원본 로그는 저장하지 않음** (선택적으로 패턴당 샘플 1~2개만 보관)

### US-3. Drift 분석
> 엔지니어로서, 배포 직후 "직전 배포 대비 무엇이 달라졌는지" 자동 분석을 보고 싶다.
- 트리거: 수동 버튼 또는 ingest 완료 후 자동
- 처리: baseline vs current 패턴/통계 비교 → drift 항목 산출 → AI 요약
- 결과: drift 리포트(신규 패턴, 사라진 패턴, 급증 패턴, 에러율 변화 등)

### US-4. 결과 확인 및 판단
> 온콜 당번으로서, drift 리포트를 보고 롤백 여부를 빠르게 판단하고 싶다.
- 화면: 배포 상세 페이지 — drift 점수, 위험도, AI 요약, 변경 패턴 목록
- 신호: `SAFE` / `WARNING` / `CRITICAL` + 롤백 권고 문구

## 4. 기능 요구사항 (Functional Requirements)

| ID | 기능 | 우선순위 | 비고 |
|----|------|---------|------|
| FR-1 | 배포 등록 API/UI | P0 | 데모 핵심 |
| FR-2 | 로그 ingest API (batch) | P0 | 패턴 추출 포함 |
| FR-3 | 패턴 정규화 (숫자/ID/타임스탬프 마스킹) | P0 | drift 정확도 핵심 |
| FR-4 | 패턴별 통계 집계 (count, error_rate 등) | P0 | |
| FR-5 | Drift 계산 (신규/소멸/급증 패턴, 지표 변화) | P0 | 데모 핵심 |
| FR-6 | AI 요약 리포트 생성 (OpenAI) | P0 | 데모 핵심 |
| FR-7 | 위험도/롤백 권고 산정 | P0 | 규칙 기반 + AI |
| FR-8 | 대시보드 (배포 목록 / 상세 / drift 뷰) | P0 | |
| FR-9 | Slack/Webhook 알림 | P2 | 시간 남으면 |
| FR-10 | 시드 데이터 / 데모용 시뮬레이터 | P1 | 데모 안정성 |

P0 = 반드시 / P1 = 가능하면 / P2 = 보너스

---

## 5. 비기능 요구사항 (Non-Functional)

- **성능**: ingest 1만 라인 배치 처리 ≤ 수 초. drift 분석 ≤ 30초(AI 호출 포함).
- **비용**: 원본 로그 미저장으로 DB 비용 최소화. OpenAI 호출은 drift당 1~2회로 제한.
- **단순성**: 단일 Next.js 앱(App Router) + Route Handlers로 백엔드 겸용. 별도 마이크로서비스 없음.
- **재현성**: 데모용 시드 스크립트로 언제든 동일한 drift 시나리오 재현 가능.

---

## 6. 성공 지표 (Demo Success Criteria)

해커톤 데모가 성공했다고 보는 기준:

1. ✅ 두 배포(정상 vs 문제) 데이터를 넣으면 drift가 자동 감지된다.
2. ✅ "신규 에러 패턴"과 "에러율 상승"이 명확히 화면에 보인다.
3. ✅ AI가 변화를 1~2문단으로 요약하고 롤백 권고를 준다.
4. ✅ 전체 흐름(배포 등록 → ingest → 분석 → 리포트)이 90초 데모 안에 들어간다.

---

## 7. 가정 및 제약

- 로그는 외부에서 LogLens로 **push** 된다고 가정(에이전트/CI에서 호출).
- 해커톤에서는 로그 소스를 직접 연동하지 않고, 시뮬레이터/시드로 대체.
- "직전 배포"를 baseline으로 자동 선택(같은 서비스·환경의 가장 최근 완료 배포).
